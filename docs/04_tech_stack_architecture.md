# 04. 기술 스택 및 아키텍처 분석

이 문서는 프로젝트의 성능, 안정성, 확장성을 위해 채택된 기술 스택과 동시성 제어 아키텍처를 상세히 분석합니다.

## 🛠️ 핵심 기술 스택

### 1. 언어 및 런타임: Rust & Tokio
- **선택 이유**: 금융 데이터 처리는 미세한 지연 시간(Latency)도 중요합니다. Rust는 가비지 컬렉션 없는 고성능을 제공하며, `tokio`는 수백 개의 WebSocket 연결을 효율적으로 처리할 수 있는 비동기 I/O를 지원합니다.
- **활용**: `tokio::spawn`을 통해 데이터 수집, DB 쓰기, 헬스 체크 등의 태스크를 독립적인 경량 스레드(Green Threads)로 실행합니다.

### 2. 시계열 데이터베이스: QuestDB
- **선택 이유**: 초당 수십만 건의 데이터를 수집하면서 동시에 복잡한 시계열 쿼리를 수행해야 합니다. QuestDB는 컬럼 지향 저장소와 In-memory 처리를 결합하여 최적의 성능을 냅니다.
- **활용**: `questdb-rs` 클라이언트를 사용하여 InfluxDB Line Protocol(ILP)을 통해 고속 벌크 인서트를 수행합니다.

### 3. 정밀 연산: `rust_decimal`
- **선택 이유**: 금융 계산에서 부동 소수점(f64)은 오차를 발생시킵니다.
- **활용**: 모든 가격, 수량, 수익률 계산에 `Decimal` 타입을 사용하여 정확한 수치를 보장합니다.

## 🔄 동시성 및 통제 아키텍처

### 1. 비동기 메시지 패싱 (MPSC)
- **방식**: `tokio::sync::mpsc` 채널을 사용하여 데이터 생산자(WS Task)와 소비자(DB Writer)를 완전히 분리(Decoupling)합니다.
- **이점**: DB 지연이 발생하더라도 채널 버퍼가 이를 완충하며, 생산자 태스크는 블로킹 없이 다음 데이터를 수집할 수 있습니다.

### 2. 우아한 종료 (Graceful Shutdown)
- **방식**: `tokio_util::sync::CancellationToken`을 사용합니다.
- **과정**: Ctrl+C 시그널 수신 시 토큰을 취소(Cancel)하면, 모든 비동기 루프가 이를 감지하여 현재 진행 중인 작업을 정리하고 종료됩니다. 특히 DB Writer는 종료 직전 버퍼에 남은 데이터를 모두 Flush하도록 보장합니다.

### 3. 에러 전파 및 관리
- **`anyhow` & `thiserror`**: 복잡한 비동기 체인에서 발생하는 에러를 체계적으로 래핑하고 컨텍스트를 추가하여 디버깅을 용이하게 합니다.
- **Tracing**: `tracing` 라이브러리를 통해 구조화된 로그를 남기며, 비동기 태스크 간의 흐름을 추적합니다.

## 🚀 성능 최적화 전략
- **Zero-Copy Deserialization**: `serde`를 활용하여 불필요한 메모리 복사를 최소화하면서 빠르게 JSON을 파싱합니다.
- **Batching Strategy**: DB 쓰기 성능을 극대화하기 위해 메모리 상에서 데이터를 일정량 모았다가 한 번에 기록합니다. 이는 디스크 I/O 횟수를 획기적으로 줄여줍니다.
- **Event Merging**: 백테스트 시 대량의 데이터를 메모리에 올리지 않고, 필요에 따라 스트리밍하거나 효율적으로 병합하여 메모리 사용량을 조절합니다.
